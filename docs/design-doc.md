## Problem Statement

Applying to college means juggling three hard things at once: figuring out *which*
schools to apply to, knowing your *honest* odds at each one, and never missing a
deadline across a dozen schools with different EA/ED/RD dates, supplements, and
financial aid forms. Students currently do this in a spreadsheet plus a pile of
browser tabs (Common App, Naviance/SCOIR, Niche, BigFuture). Nothing ties it
together and nothing tells them the truth about whether their list is a good bet.

Goal: a free web app, useful for everyone, that does the complete job — build a
list, grade its balance, estimate honest admission chances, compare schools
side by side, and track every deadline.

## What Makes This Cool

The spine is an honest **list-health engine**: "Your list is 8 reaches, 1 target,
1 safety. That's a bad bet. Here's what to add." Every big site shows you data;
none of them tell you your list is broken and what to do about it. That grade is
the connective tissue between the two jobs (build the list / run the list) and the
one thing a stressed 17-year-old can't get anywhere else.

Second distinctive bet: rankings by *what the student cares about* (net price after
aid, grad rate, earnings, fit for their stats and major) instead of copying one
magazine's formula.

## Constraints

- **Free for users.** No paywall. Success is "students actually open it instead of
  their spreadsheet," not revenue.
- **Honesty is non-negotiable.** Admission chances are labeled rough estimates, never
  fake percentages like "73%."
- **US-focused at v1** (College Scorecard is US-only).
- **Owner maintains deadline data** each application cycle — accepted as ongoing cost.
- Side project. Should be buildable and maintainable by one person with AI leverage.
- **UI must look intentional, not AI-generated.** No generic dashboard look, no
  purple gradients, no centered-card-on-gray-background slop, no emoji-as-icons. Aim
  for a real product feel: a distinctive but restrained type system, a calm palette,
  generous whitespace, clear information hierarchy, and considered empty/loading/error
  states. The list-health grade is the visual centerpiece and should feel designed,
  not bolted on. Build the frontend with the `/frontend-design` skill, and review it
  with `/design-review` before shipping.

## Premises (all confirmed by user)

1. One product, two connected jobs (build a list / run a list), unified by the
   list-health engine. **Agreed.**
2. The hard part is data, not UI. Admission rate, SAT/ACT middle 50%, net price by
   income, majors, and outcomes come free from the **College Scorecard API** + IPEDS.
   **Agreed.**
3. Deadlines have no free API. Revised plan (user request): use a **semi-automated
   pipeline** — fetch each school's admissions page + Common App member page, extract
   dates with an LLM, and route them to a human review queue before publishing. Full
   automation is rejected on purpose: a wrong deadline is the worst possible failure
   and breaks the honesty rule, so a human approves every date. This cuts maintenance
   ~80% vs hand entry without risking a student's application. **Agreed (revised).**
4. Don't copy US News rankings (copyrighted, paywalled, least useful). Derive
   "rank by what you care about" from Scorecard outcomes instead. **Agreed.**
5. "Chances of admission" is an honest stats-based estimate (your GPA/scores vs the
   school's middle 50% → reach/target/safety), transparently labeled. **Agreed —
   "be honest."**

## Approaches Considered

### Approach A: Local-first SPA, no backend
Next.js + Scorecard API + JSON deadlines + localStorage. Effort S, Risk Low.
Ships fast and free, but no accounts, sync, or sharing.

### Approach B: Full-stack app with accounts (CHOSEN)
Next.js full-stack + Supabase (Postgres, Auth, RLS). Server caches Scorecard
nightly; admin UI edits deadlines without redeploy; accounts give cross-device
sync and shareable lists. Effort L, Risk Med.

### Approach C: Local-first + automated data pipeline
SPA with a scheduled GitHub Action refreshing a shipped static dataset; sharing via
URL-encoded links; no DB/auth. Effort M, Risk Low. Complete features without a
backend, but no true multi-device sync.

## Recommended Approach

**Approach B**, chosen by the user under the "boil the ocean" principle: build the
complete product, including accounts, sync, and sharing, rather than deferring them.
Reasoning the user accepted: a real product that scales to "everyone" needs server-
cached data (easy refresh, no client-side dataset bloat), an admin editor for the
deadline data, and accounts so a student's list follows them across devices and can
be shared read-only with a parent or counselor.

### Architecture sketch
- **Frontend:** Next.js (App Router) + TypeScript + Tailwind + shadcn/ui.
- **Backend/data:** Supabase — Postgres, Auth, Row Level Security. Deploy on Vercel
  (auto-deploy on merge). Scheduled job (Vercel Cron or Supabase scheduled function)
  refreshes the Scorecard cache nightly/weekly.
- **Data sources:** College Scorecard API (free key from api.data.gov) as primary;
  IPEDS to fill gaps. Cached into a `colleges` table — never hit live per request.
- **Deadlines:** `college_deadlines` table fed by a **semi-automated extraction
  pipeline** — a scheduled job fetches each school's admissions page + Common App
  member page, an LLM (Claude) extracts structured dates, and results land in an
  owner-only **review queue**. The owner approves/edits each record before it goes
  live; nothing publishes unverified. Source URL and last-verified date are stored
  per record and shown in the UI so students can see where a date came from.
- **Email reminders:** Resend (or Supabase) for "due this week" nudges.

### Core features
1. **Discovery & suggestions** — search/filter by major, location, size, selectivity,
   net price; "schools like X"; suggestions to fill gaps in a thin list.
2. **List management** — add/remove, per-school status (researching / applying /
   submitted / accepted / etc.), notes.
3. **List-health engine** — pure function: student stats + each school's stats →
   reach/target/safety bucket → overall balance grade + concrete "add a safety"
   guidance. The spine; has no UI dependency, so build it first.
4. **Honest chancing** — stats-based estimate per school, labeled as an estimate with
   the data source and year shown. No fabricated percentages.
5. **Side-by-side comparison** — net price by income band, admit rate, test ranges,
   target major strength, grad rate, earnings outcomes.
6. **Deadline tracker** — per-school EA/ED/RD, supplement essays, FAFSA/CSS; a "due
   this week" view; email reminders.
7. **Accounts + sharing** — sync across devices; read-only share link for parents/
   counselors.

### Data model (first cut)
- `users` (Supabase auth)
- `student_profiles` (gpa, sat, act, intended_major, location/size/cost prefs)
- `colleges` (Scorecard-sourced: admit_rate, sat/act p25/p75, net_price_by_income,
  majors, grad_rate, earnings, source_year)
- `college_deadlines` (college_id, term, ed/ea/rd dates, fafsa/css, supplements)
- `user_lists` / `list_items` (college_id, status, notes, deadline overrides)
- `shares` (list_id, token, read-only)

## Open Questions

- Refresh cadence for Scorecard cache — nightly vs weekly (weekly is plenty; the data
  changes annually).
- Reminder channel at v1 — email only, or add browser push later?
- International students — out of scope at v1 (Scorecard is US-only); flag clearly so
  they aren't misled.
- Seed data: which ~50-100 schools to curate deadlines for first (cover the most-
  applied-to set before going long-tail).
- Chancing model honesty: how to present "we don't have enough data for this school"
  rather than guessing.

## Success Criteria

A real student, mid-application, can:
1. Build a complete list from scratch using discovery + suggestions.
2. See an honest reach/target/safety grade and act on it (adds a safety because the
   tool told them to).
3. Compare two schools on net price after aid, not sticker price.
4. Never miss a deadline — the "due this week" view is correct for every school on
   their list.
5. Share the list with a parent via link.

The real bar: a student uses this *instead of* their spreadsheet, and a second student
they told about it does too.

## Distribution Plan

Web service on Vercel. Auto-deploy on merge to main (existing deployment pipeline
covers this). No app store, no binary. Supabase hosts the database and auth.

## Dependencies

- College Scorecard API key (free, api.data.gov).
- Supabase project (free tier) + Vercel project.
- Curated seed deadline dataset for the first batch of schools (owner-provided).

## Next Steps (build order)

1. **Get the data flowing.** Register a Scorecard API key, define the `colleges`
   schema, write the fetch+normalize job, and load the dataset into Supabase. Prove
   you can query "schools with admit rate < 20% offering CS in the Northeast."
2. **Build the list-health engine as a pure, tested function** (student stats + school
   stats → bucket + balance grade). No UI, no DB. This is the spine and the
   differentiator — get it right and honest before anything visual.
3. Scaffold Next.js + Supabase auth + the colleges search/browse UI.
4. List management + the list-health grade rendered on the list.
5. Side-by-side comparison view.
6. Deadlines: build the extraction pipeline (fetch admissions/Common App pages →
   LLM extract → review queue) + the owner approval UI + the "due this week" view +
   email reminders. Seed and verify the most-applied-to schools first.
7. Share links.
8. Design pass: run `/frontend-design` for the build and `/design-review` before
   ship to kill any generic-AI-looking patterns.

## The Assignment

Before writing a line of code: find 3 actual students (or this-year applicants) who
are mid-process and **watch** them work their current list — spreadsheet, Common App,
Naviance, whatever. Don't demo, don't help. Two specific things to capture:

1. Point at their list and ask, "Which of these are reaches?" Watch whether they
   actually know. If they hesitate or guess wrong, the list-health grade is your
   killer feature. If they answer instantly and correctly, it's a nice-to-have and the
   deadline tracker is your real wedge. This decides what you build first.
2. Get the *exact* deadline data (EA/ED/RD, supplements, FAFSA/CSS) for the ~20 schools
   those three students care about. That's your seed dataset and your test cases — and
   it tells you how painful the curation really is.

# Engineering Review (plan-eng-review, 2026-06-15)

## Locked decisions

- **D1 — Phased build (spine first).** Full scope, sequenced into 4 shippable phases.
  Nothing cut.
- **D2 — Chancing model.** v1 buckets on admission rate, refined by the student's test
  scores vs the school's reported SAT/ACT range *when both exist*. Test-optional /
  no-range schools fall back to admit-rate-only and the UI states the basis. GPA is
  collected for the student's own reference but NOT scored against school data (College
  Scorecard has no admitted-GPA field). Community scattergram (self-reported
  admit/reject outcomes) is the long-term accuracy layer, added once there are users.
- **D3 — Security model, complete at the first auth commit.** RLS on every user table
  keyed to `auth.uid()`, with a "user B cannot read user A's list" test. Share links use
  random, revocable, read-only tokens (never list ids). Age gate at signup, posted
  privacy policy, data minimization (store only GPA/scores/list).

## Phase plan

```
PHASE 1 — SPINE (prove the differentiator)
  Scorecard ingestion job ──► colleges table ──► search + list UI
                                    │
  list-health engine (pure) ────────┘──► reach/target/safety grade on the list
  (local persistence, pre-accounts)

PHASE 2 — COMPARE + CHANCE
  side-by-side comparison view  |  per-school chancing badge w/ honest fallbacks

PHASE 3 — DEADLINES (the one innovation token)
  seed top ~50 schools by hand ──► validate UX ──► LLM extraction + review queue
  "due this week" view  |  email reminders  |  store source URL + last-verified date

PHASE 4 — ACCOUNTS
  Supabase auth + full RLS + isolation test  |  cross-device sync
  random revocable share links  |  age gate + privacy policy
```

## Baked-in recommendations (no decision needed)

- **Deadline pipeline = your single innovation token.** Spend it deliberately. Seed the
  most-applied-to ~50 schools manually in Phase 3, confirm the UX, *then* add LLM
  extraction. Every record stores its source URL and last-verified date, shown in the UI.
- **DRY:** one `classifyAdmissionChance(studentStats, collegeStats)` module is the single
  source of truth for both the list-health grade and the per-school chancing badge. Do not
  duplicate the bucketing logic in two places.
- **Search:** Postgres full-text or `pg_trgm` index on college name + filter columns. Never
  scan ~6k rows per keystroke.
- **List view:** fetch list items joined to colleges in one query. No per-item lookups
  (N+1).
- **Scorecard ingest:** batched, paginated, upsert, in the scheduled job only. Never call
  the Scorecard API live on a user request — always serve the cached table with a "data as
  of {year}" label.
- **Outcomes honesty:** earnings/grad-rate come from Scorecard with a cohort-year lag.
  Always label the year so a number is never presented as current-truth.

## Failure modes

| Codepath | Realistic failure | Test? | Error handling? | User sees |
|---|---|---|---|---|
| chancing w/ missing scores | shows empty/garbage bucket | required | required fallback | honest "based on selectivity" label |
| Scorecard ingest | API down mid-job | required | serve stale + "data as of" | no breakage |
| deadline record | stale/wrong date | review queue + eval | last-verified shown | **catastrophic if silent — gated** |
| RLS misconfig | reads another user's list | isolation test | DB-enforced RLS | **silent leak if untested — gated** |
| share token | guessable URL | E2E | random token | leak if weak — gated |

Critical gaps if shipped without the marked test AND error handling: deadline accuracy
and RLS isolation. Both are gated by the locked decisions above.

# Design Review (plan-design-review, 2026-06-15)

Main screen rated 3/10 → 7/10. Mockups saved under

## Locked design decisions

- **Verdict tone: blunt but kind.** Truth on line one, the fix on line two:
  "8 reaches, 1 target, 1 safety. This is risky — let's add a safety or two."
  Shown with the count, a balance bar (coral/amber/teal), and a single CTA
  "Add a safety school." It's the first thing on the page; it IS the product.
- **Row model (revised by user).** Drag-to-reorder rows. Columns: School |
  Importance (student-set, **5-star**) | **Tier** (the bucket — Reach / Target /
  Likely / Safety, its own column) | **Projected odds (est.)** (a separate column,
  honest `~%` estimate) | Next deadline. Tier and projected odds are deliberately
  separate: tier is the qualitative bucket, projected odds is the number.
  **Net price removed from the row** — it moves to the comparison view.
- **Four tiers, not three.** Reach (coral) / Target (amber) / Likely (blue) /
  Safety (teal). The verdict and balance bar count all four. "Likely" is the real
  category between target and safety.
- **Projected admit % = honest estimate only.** Display as `~12% est` or a range,
  with the basis available on tap. Never a bare exact percentage. This reconciles
  the user's request for a number with the eng-review honesty rule (D2). Reaches
  read low (~5%), safeties high (~78%), and schools with thin data show a wider
  range or admit-rate-only.
- **Reach/target/safety encoded by color AND label** (colorblind-safe). Coral =
  reach, amber = target, teal = safety.

## Screen specs

**My list (main) — mockup approved.**
- Empty state (new user, zero schools): not "no schools found." A warm "Start your
  list" with a search box plus a "not sure where to start? answer 3 questions" path.
  The empty state IS the onboarding.
- Loading: skeleton rows, never a bare spinner.
- Mobile: rows reflow to one card per school (name, importance, odds tag + estimate,
  deadline stacked); verdict pinned at top; CTA full-width; 44px touch targets;
  reorder via long-press or up/down controls (no hover-only affordances).

**Comparison view ("why this school", side-by-side).**
- 2-3 schools as columns; rows are the factors that actually decide it: estimated
  chance (honest), admit rate, the school's test range vs the student's scores,
  target-major strength, grad rate + earnings (cohort-year labeled), and **net price
  by income band** (net price lives here now).
- Subtly mark the better value per row — no gaudy "winner" badge.
- One-school state: prompt to add a second school to compare.

**Deadlines view.**
- Default to "due this week / next," grouped by urgency — not a static full calendar.
- Each item shows source + last-verified date (trust) and links to the app page.
- Past-due vs upcoming visually distinct; nothing silently disappears.

**Decision-analysis features (user-requested — THREE distinct things).**

**1. Weighted decision matrix (its own page).**
- A **radar / spider chart** (styled like the user's reference image) of the decision
  factors a student weighs: Academics, Major strength, Career outcomes, Financial aid,
  Affordability, Location, Campus life, Selectivity (factor set tunable).
- **Importance sliders, 0–5 per factor.** Dragging a slider reshapes the radar live.
  The shape is the student's priority profile.
- Each school is scored 0–5 per factor; the matrix produces a **weighted fit score**
  per school = Σ(factor_score × importance) / Σ(importance). Optionally overlay a
  school's scores as a second polygon over the priority polygon ("what you want" vs
  "what this school offers").
- Mockup approved inline (interactive radar + sliders).
- States: factors default to sensible weights; empty school = show priority profile
  only; persist the student's weights to their profile.

**2. Plot vs importance to you (chart).**
- Schools plotted on an importance axis (the 5-star personal grade) against their
  weighted fit score, so a student sees which high-importance schools actually fit.

**3. Plot vs projected acceptance rate (separate chart).**
- Schools plotted on a projected-acceptance-rate axis (honest estimate — same source as
  the row's projected odds, never a fake exact %) against weighted fit / importance.
- Quadrant reading still applies: high-want + high-odds = strong bets; an empty
  high-want/high-odds region is visual proof the list is top-heavy, reinforcing the
  list-health verdict.
- States for 2 & 3: need ≥2 schools + scores to plot; schools missing data go in a
  muted "not enough data to place" tray, never silently dropped.

All three live in Phase 2 (the "reason about the list" views), built on the same
chancing + importance + weighted-fit data.
